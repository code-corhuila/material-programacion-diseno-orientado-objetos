Semana 1: Fundamentos y planificación de proyectos móviles

En esta primera semana construiremos el “mapa del territorio”: qué es una app móvil, cómo se organiza un proyecto, qué decisiones clave se toman al inicio y cómo planificar un MVP (producto mínimo viable) con enfoque ágil.

1. Fundamentos del desarrollo móvil

Antes de programar pantallas o conectar servicios, necesitamos un lenguaje común sobre apps móviles, plataformas, frameworks, APIs y ciclo de vida de desarrollo.

Aplicación móvil: software diseñado para ejecutarse en dispositivos móviles (Android/iOS), que interactúa con hardware (sensores, cámara, GPS) y servicios (internet, APIs).
Proyecto móvil: proceso planificado para construir una app (producto) con objetivos, alcance, entregables, recursos y fechas, usando una metodología (ágil o tradicional).

Un proyecto móvil normalmente combina varios componentes: interfaz (UI), lógica (negocio), consumo de servicios (APIs), almacenamiento local y publicación en tiendas (release).

1.1. Componentes típicos de una app móvil

Componente ¿Qué hace? Ejemplo en una app universitaria
Interfaz (UI) Pantallas, navegación, formularios, diseño responsive. Login, listado de eventos, detalle del evento.
Lógica de negocio Reglas, validaciones, estado, flujos. Validar cupos, confirmar inscripción, generar ticket.
Consumo de APIs Comunicación con backend (REST/GraphQL) y servicios externos. Consultar eventos, registrar asistencia, notificar cambios.
Almacenamiento local Persistencia offline/caché (SQLite, preferencias, archivos). Guardar sesión, eventos favoritos, datos sin conexión.
Release y distribución Empaquetado, firmas, versiones, publicación. Subir APK/AAB o IPA a tienda, notas de versión.

1.2. Roles clave en un proyecto móvil

Equipo de producto

  • Product Owner / Cliente: define objetivos, alcance y prioridades.
  • UX/UI: diseña flujos, prototipos, interacción y accesibilidad.
  • Enfoque: valor para el usuario, claridad del alcance, experiencia.

Equipo técnico

  • Desarrollo: construye la app, integra APIs, maneja estados y pantallas.
  • QA: prueba funcionalidades, rendimiento, compatibilidad y errores.
  • Enfoque: calidad, estabilidad, seguridad y entrega continua.

2. Ecosistema móvil: nativo, híbrido y PWA

Una de las primeras decisiones del proyecto es el enfoque tecnológico. No hay una “mejor” opción universal: depende del presupuesto, tiempos, equipo y necesidades (GPS, cámara, rendimiento, offline, etc.).

Aspecto Nativo (Android/iOS) Híbrido (Flutter/React Native/Ionic)
Rendimiento Muy alto (acceso directo a SDKs). Alto (depende del framework y del caso).
Tiempo de desarrollo Mayor (2 bases de código si es multi-plataforma). Menor (una base de código para Android e iOS).
Acceso a hardware Completo y más estable. Muy bueno; a veces requiere plugins/bridges.
Mantenimiento Separado por plataforma. Centralizado en gran medida.
Casos recomendados Apps críticas, alto rendimiento, UX nativa específica. Apps de negocio, prototipos rápidos, equipos web.

3. Planificación de proyectos móviles (enfoque ágil)

Planificar no es “adivinar el futuro”, sino crear un plan flexible que permita entregar valor pronto y aprender rápido. En móvil, el alcance suele crecer, por eso el MVP y el backlog son esenciales.

3.1. Artefactos mínimos para iniciar bien

Artefacto ¿Para qué sirve? Ejemplo (App universitaria)
Objetivo del producto Alinea al equipo con el “para qué”. Facilitar inscripción y asistencia a eventos con QR.
Alcance + fuera de alcance Evita “meter de todo” desde el inicio. Incluye: eventos + QR. No incluye: pagos en la semana 1.
User stories Define necesidades desde el punto de vista del usuario. “Como estudiante, quiero ver eventos por fecha para planear.”
Prototipo (wireframe) Valida flujo antes de programar. Login → Home → Detalle evento → Inscripción.
MVP Primera versión funcional con valor real. Ver eventos + inscribirse + QR de asistencia.

3.2. Estrategias de planificación (rápidas y útiles)

Planificación ágil (recomendada)
  • Backlog: lista priorizada de historias.
  • Sprints: entregas cortas (1–2 semanas).
  • Revisión: mostrar avances y ajustar.
  • Métrica: valor entregado + feedback.
Control de riesgos desde el inicio
  • Riesgo técnico: cámara/GPS/offline → prototipo temprano.
  • Riesgo de alcance: definir “fuera de alcance”.
  • Riesgo de tiempo: MVP realista + priorización MoSCoW.
  • Riesgo de calidad: pruebas básicas por feature.

Actividad práctica de la semana

1. Contexto

Durante el curso desarrollarás un proyecto móvil para entregar al final. En esta primera semana no vas a programar todavía: vas a definir la base del proyecto para evitar problemas de alcance, falta de claridad y cambios innecesarios más adelante.

  1. Elige una idea de aplicación (problema real + usuarios claros).
  2. Define el MVP (mínimo funcional viable) y lo que queda fuera de alcance.
  3. Identifica riesgos iniciales y una ruta de trabajo simple por semanas.

2. Entregables

Entregable Qué incluye Formato sugerido
Mapa del Proyecto (1 página) Problema, usuarios, objetivo, propuesta de valor y alcance general. PDF / Documento / Markdown
MVP + Fuera de alcance 3–6 funcionalidades del MVP + 3 funcionalidades que NO se harán aún. Lista / Tabla
Top 5 funcionalidades Priorización simple (Must/Should/Could) para orientar el trabajo. Lista
Riesgos iniciales 3 riesgos (técnicos o de alcance) + cómo los mitigarías. Tabla
Ruta de trabajo (muy básica) Plan de 4 semanas: qué lograrás cada semana (alto nivel). Tabla / Lista
